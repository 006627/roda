Instalar novos formatos de metadados descritivos

O RODA suporta qualquer formato de metadados descritivo (i.e. Informação Descritiva conforme indicado no OAIS) desde que representado por um ficheiro XML. Se tem um formato de metadados descritivo que não é baseado em XML (por exemplo, CSV, JSON, MARC21, etc.), terá que converter para XML antes de poder usar no RODA. Existem várias ferramentas na Web que permitem converter a maioria dos formatos de dados em XML.

Depois de ter os metadados em XML, está pronto para empacotá-lo num Pacote de Informações de Envio (SIP) e ingeri-lo no repositório. Como alternativa, você pode criar um arquivo de metadados diretamente no repositório usando a funcionalidade fornecida pelo Catálogo. Quando o formato de metadados é novo para o RODA, o repositório fará o seu melhor para suportar sem a necessidade de fazer qualquer reconfiguração do sistema, no entanto, as seguintes limitações se aplicam:

Validação

Se nenhum schema for fornecido para o formato de metadados, o repositório verificará se o arquivo XML de metadados está bem formado, mas como o repositório não tem noção da gramática correta, ele não verificará se o arquivo é válido.

Indexação

O repositório irá indexar todos os elementos textuais e valores de atributos encontrados no ficheiro de metadados. No entanto, uma vez que o repositório não conhece o mapeamento correcto entre os elementos XML e o modelo de dados interno, apenas a pesquisa básica será possível nos metadados fornecidos.

Visualização

Quando nenhum mapeamento de visualização é configurado, um visualizador de metadados genérico será usado pelo repositório para exibir os metadados baseados em XML. Todos os elementos textuais e atributos serão mostrados com nenhuma ordem específica e seu XPath será usado como rótulo.

Edição

O RODA precisa de um arquivo de configuração para informar como os arquivos de metadados devem ser exibidos para fins de edição. Se não existir tal configuração, o repositório exibirá uma área de texto onde o usuário é capaz de editar o XML diretamente.

Para suportar novos formatos de metadados, o repositório deve ser configurado de acordo com isso. As secções a seguir descrevem com detalhe o conjunto de ações que precisam ser executadas para suportar totalmente um novo esquema de metadados no RODA.

Arquivos de melhoramento de metadados

Para melhorar a experiência de metadados no RODA, há 4 arquivos que precisam ser adicionados às pastas de configuração do repositório. As secções a seguir descrevem e fornecem exemplos de tais arquivos.

Validação

O RODA usa um schema XML para validar a estrutura e os tipos de dados do arquivo de metadados fornecido. O esquema de Validação será usado durante o processo de ingestão para verificar se os metadados incluídos no SIP são válidos de acordo com as restrições estabelecidas, bem como quando os metadados são editados através do catálogo.

Pode-se usar um ficheiro de schema padrão para fins de validação ou criar um específico que verifique todas as condições particulares que se precisa verificar na instalação de repositório, e.g. Campos obrigatórios, vocabulários fechados para os valores de certos elementos, etc.

Os arquivos do schema de Validação devem ser colocados na pasta [RODA_HOME]/config/schemas/.

A maioria dos formatos de metadados são publicados juntamente com a documentação e um schema XML. O exemplo a seguir representa o esquema para o formato de metadados Simple Dublin Core: 

    
<?xml version="1.0" encoding="UTF-8"?>
<xs:schema  xmlns:xs="http://www.w3.org/2001/XMLSchema"
            xmlns:dc="http://purl.org/dc/elements/1.1/"
            elementFormDefault="qualified"
            attributeFormDefault="unqualified">

  <xs:import namespace="http://www.w3.org/XML/1998/namespace"
    schemaLocation="xml.xsd">
  </xs:import>

  <xs:complexType name="SimpleLiteral">
    <xs:complexContent mixed="true">
      <xs:restriction base="xs:anyType">
        <xs:sequence>
          <xs:any processContents="lax" minOccurs="0" maxOccurs="0"/>
        </xs:sequence>
        <xs:attribute ref="xml:lang" use="optional"/>
      </xs:restriction>
    </xs:complexContent>
  </xs:complexType>

  <xs:element name="any" type="SimpleLiteral" abstract="true"/>

  <xs:element name="title" substitutionGroup="any"/>
  <xs:element name="creator" substitutionGroup="any"/>
  <xs:element name="subject" substitutionGroup="any"/>
  <xs:element name="description" substitutionGroup="any"/>
  <xs:element name="publisher" substitutionGroup="any"/>
  <xs:element name="contributor" substitutionGroup="any"/>
  <xs:element name="date" substitutionGroup="any"/>
  <xs:element name="type" substitutionGroup="any"/>
  <xs:element name="format" substitutionGroup="any"/>
  <xs:element name="identifier" substitutionGroup="any"/>
  <xs:element name="source" substitutionGroup="any"/>
  <xs:element name="language" substitutionGroup="any"/>
  <xs:element name="relation" substitutionGroup="any"/>
  <xs:element name="coverage" substitutionGroup="any"/>
  <xs:element name="rights" substitutionGroup="any"/>

  <xs:group name="elementsGroup">
    <xs:sequence>
      <xs:choice minOccurs="0" maxOccurs="unbounded">
        <xs:element ref="any"/>
      </xs:choice>
    </xs:sequence>
  </xs:group>

  <xs:complexType name="elementContainer">
    <xs:choice>
      <xs:group ref="elementsGroup"/>
    </xs:choice>
  </xs:complexType>

   <xs:element name="simpledc" type="elementContainer"/>
</xs:schema>

Indexação

A atividade de Indexação é suportada por uma XSLT (eXtensible Stylesheet Language Transformations) que transforma os metadados baseados em XML em algo que o mecanismo de indexação é capaz de entender. Esse ficheiro é responsável por selecionar os dados que devem ser indexados, mapear dados em nomes de campos específicos e instruir o mecanismo sobre como os dados devem ser indexados com base no seu tipo de dados (i.e. texto, número, data etc.).

O ficheiro com o mapa de índice deve ser adicionado na pasta [RODA_HOME]/config/crosswalks/ingest/.

O seguinte é um exemplo de um mapa de índice para o exemplo Simple Dublin Core.

  exclude-result-prefixes="dc">
  <xsl:output method="xml" indent="yes" encoding="UTF-8"
    omit-xml-declaration="yes" />

  <xsl:template match="/">
    <doc>
      <xsl:apply-templates />
    </doc>
  </xsl:template>
  <xsl:template match="simpledc">
    <xsl:if test="count(title)  &gt; 0">
      <xsl:if test="title[1]/text()">
        <field name="title">
          <xsl:value-of select="title[1]/text()" />
        </field>
      </xsl:if>
      <xsl:for-each select="title">
        <xsl:if test="normalize-space(text())!=''">
          <field name="title_txt">
            <xsl:value-of select="text()" />
          </field>
        </xsl:if>
      </xsl:for-each>
    </xsl:if>
    <xsl:if test="count(description)  &gt; 0">
      <xsl:if test="description[1]/text()">
        <field name="description">
          <xsl:value-of select="description[1]/text()" />
        </field>
      </xsl:if>
      <xsl:for-each select="description">
        <xsl:if test="normalize-space(text())!=''">
          <field name="description_txt">
            <xsl:value-of select="text()" />
          </field>
        </xsl:if>
      </xsl:for-each>
    </xsl:if>
    <xsl:for-each select="contributor">
      <xsl:if test="normalize-space(text())!=''">
        <field name="contributor_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="coverage">
      <xsl:if test="normalize-space(text())!=''">
        <field name="coverage_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="creator">
      <xsl:if test="normalize-space(text())!=''">
        <field name="creator_txt">
          <xsl:value-of select="text()" />

      </xsl:if>

            <xsl:value-of select="regex-group(0)" />
          </xsl:variable>
          <xsl:if test="not(normalize-space($date)='')">
            <field name="dateInitial">
              <xsl:value-of select="$date" />
              <xsl:text>T00:00:00Z</xsl:text>
            </field>
            <field name="dateFinal">
              <xsl:value-of select="$date" />
              <xsl:text>T00:00:00Z</xsl:text>
            </field>
          </xsl:if>
        </xsl:matching-substring>
      </xsl:analyze-string>
      </xsl:if>
      <xsl:if test="count(date)  &gt; 1">
        <xsl:analyze-string regex="^\d{{4}}-\d{{2}}-\d{{2}}$" select="date[1]/text()">
          <xsl:matching-substring>
            <xsl:variable name="date">
              <xsl:value-of select="regex-group(0)" />
            </xsl:variable>
            <xsl:if test="not(normalize-space($date)='')">
              <field name="dateInitial">
                <xsl:value-of select="$date" /><xsl:text>T00:00:00Z</xsl:text>
              </field>
            </xsl:if>
          </xsl:matching-substring>
        </xsl:analyze-string>
        <xsl:analyze-string regex="^\d{{4}}-\d{{2}}-\d{{2}}$" select="date[2]/text()">
          <xsl:matching-substring>
            <xsl:variable name="date">
              <xsl:value-of select="regex-group(0)" />
            </xsl:variable>
            <xsl:if test="not(normalize-space($date)='')">
              <field name="dateFinal">
                <xsl:value-of select="$date" />
                <xsl:text>T00:00:00Z</xsl:text>
              </field>
            </xsl:if>
          </xsl:matching-substring>
        </xsl:analyze-string>
      </xsl:if>
      <xsl:for-each select="date">
        <xsl:if test="normalize-space(text())!=''">
          <field name="date_txt">
            <xsl:value-of select="text()" />
          </field>
        </xsl:if>
      </xsl:for-each>
    </xsl:if>

    <xsl:for-each select="format">
      <xsl:if test="normalize-space(text())!=''">
        <field name="format_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="identifier">
      <xsl:if test="normalize-space(text())!=''">
        <field name="identifier_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="language">
      <xsl:if test="normalize-space(text())!=''">
        <field name="language_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="publisher">
      <xsl:if test="normalize-space(text())!=''">
        <field name="publisher_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="relation">
      <xsl:if test="normalize-space(text())!=''">
        <field name="relation_txt">

        <field name="rights_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="source">
      <xsl:if test="normalize-space(text())!=''">
        <field name="source_txt">

        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="subject">
      <xsl:if test="normalize-space(text())!=''">
        <field name="subject_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <xsl:for-each select="type">
      <xsl:if test="normalize-space(text())!=''">
        <field name="type_txt">
          <xsl:value-of select="text()" />
        </field>
      </xsl:if>
    </xsl:for-each>
    <field name="level">item</field>
  </xsl:template>
</xsl:stylesheet>

O resultado produzido por esta folha de estilo é um documento Solr pronto para ser indexado pelo mecanismo de pesquisa RODA. Veja o exemplo abaixo: 

    
<doc>

  <field name="title_txt">{{title}}</field>

Edição

A atividade de Edição é suportada por um ficheiro de configuração que instruirá o repositório sobre como exibir um formulário para editar metadados existentes. O ficheiro de configuração também serve para fornecer um modelo para criar um novo item de metadados com alguns atributos predefinidos já preenchidos.

Os templates de formulário devem ser adicionados à configuração na pasta [RODA_HOME]/config/templates/. O exemplo a seguir mostra como um ficheiro de template pode ser combinado com anotações que serão usadas para processar o editor de metadados. 

    
{{~field name="title"   order='2' auto-generate='title' label="{'en': 'Title'}" xpath="//*:title/string()"}}
{{~field name="id"      order='1' auto-generate='id' label="{'en': 'ID'}" xpath="//*:identifier/string()"}}
{{~field name="creator"   label="{'en': 'Creator'}" xpath="//*:creator/string()"}}
{{~field name="dateInitial" order='3' type='date' label="{'en': 'Initial date'}" xpath="//*:date[1]/string()"}}
{{~field name="dateFinal"   order='4' type='date' label="{'en': 'Final date'}" xpath="//*:date[2]/string()"}}
{{~field name="description" type='text-area' label="{'en': 'Description'}" xpath="//*:description/string()"}}
{{~field name="producer"  label="{'en': 'Producer'}" xpath="//*:publisher/string()"}}
{{~field name="rights"    label="{'en': 'Rights'}" xpath="//*:rights/string()"}}
{{~field name="language"  auto-generate='language' label="{'en': 'Language'}" xpath="//*:language/string()"~}}
<?xml version="1.0"?>

<simpledc xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:noNamespaceSchemaLocation="../schemas/dc.xsd">
  <title>{{title}}</title>
  <identifier>{{id}}</identifier>
  <creator>{{creator}}</creator>
  <date>{{dateInitial}}</date>
  <date>{{dateFinal}}</date>
  <description>{{description}}</description>
  <publisher>{{producer}}</publisher>
  <rights>{{rights}}</rights>
  <language>{{language}}</language>
</simpledc>

Os arquivos de template de formulário baseiam-se no poderoso mecanismo Handlebars engine. Cada campo que é esperado para ser mostrado no editor de metadados deve ser identificado no início do arquivo por um campo identificador (e.g. {{~ field name = "title" ~}} ). Há várias opções que podem ser usadas para modificar a maneira como cada campo é exibido. Estas opções são pares de chave-valor, e.g. label="Título do trabalho", onde a chave é o nome da opção e o valor é o valor que será dado a essa opção.

As opções disponíveis que alteram o comportamento dos campos são:

  • order - A ordem pela qual o campo deve ser exibido no editor de metadados
  • label - O rótulo do campo a ser mostrado à esquerda do valor no editor. O formato é um mapa JSON. As chaves do mapa são o ID do idioma. Exemplo: {"en": "Title", ""pt_PT": "Título"}"
  • labeli18n - A chave i18n para o rótulo do campo (consulte a secção "Internacionalização de texto" abaixo)
  • description - Um texto com alguma ajuda relacionada com o campo
  • value - O valor predefinido do campo
  • mandatory - Se definido como verdadeiro o rótulo é desenhado em negrito para chamar a atenção
  • hidden - Se definido como verdadeiro, o campo está oculto
  • xpath - O xpath do documento XML de metadados para o qual este campo deve vincular
  • auto-generate - Preenche o valor com um dos geradores de valor automático disponíveis. Substitui o valor:
    • now - A data atual no formato ano/mês/dia
    • id - Gera um identificador
    • title - Gera um título
    • level - Adiciona o nível de descrição atual
    • parentid - Adiciona o identificador do pai, se ele existir
    • language - Adiciona o idioma do sistema, com base no locale. Exemplo: "português" or "English"
  • type - O tipo do campo. Os valores possíveis são:
    • text - campo textual
    • text-area - Área de texto. Maior que um texto de campo.
    • date - Campo de texto com um seletor de data
    • list - Lista com os valores possíveis (caixa de combinação)
      • options - Lista com os possíveis valores que um campo pode ter. Esta lista é um JsonArray. Exemplo: options="[\"final\",\"revised\",\"draft\"]"
      • optionsLabels - Mapa com os rótulos para cada opção. A chave deve corresponder a uma das opções especificadas na lista "options". A entrada é outro mapa, mapeando um idioma (chave) para um rótulo (valor). Exemplo: optionsLabels="{\"final\": {\"en\":\"Final\", \"pt_PT\":\"Final\"},\"revised\": {\"en\":\"Revised\", \"pt_PT\":\"Verificado\"},\"draft\": {\"en\":\"Draft\", \"pt_PT\":\"Rascunho\"}}"
      • optionsLabelI18nKeyPrefix - I18n prefixo chave. Todas as chaves que começam com o prefixo são usadas para construir a lista. Exemplo:
        optionsLabelI18nKeyPrefix="crosswalks.dissemination.html.ead.level"

      • Exemplo completo de um campo "list": 
        {{~field
name="statusDescription"
order="470"
type="list"
value="final"
options="[\"final\",\"revised\",\"draft\"]"

O seguinte é um exemplo de como as tags podem ser usadas: 

{{~file name="title" order="1" type="text" label="Template title" mandatory="true" auto-generate="title"~}}

Ativar o novo formato

Depois de adicionar todos os arquivos descritos na seção anterior, é preciso habilitá-los no repositório. Para realizar isso, as atividades a seguir precisam ser feitas.

Ativar o novo formato de metadados

Depois de adicionar os arquivos descritos anteriormente à sua pasta de configuração, você deve ativar o novo formato no arquivo de configuração principal do RODA.

Edite o ficheiro [RODA_HOME]/config/roda-wui.properties e adicione uma nova entrada como as mostradas no exemplo a seguir com o nome do seu formato de metadados recentemente adicionado. Isso fará RODA ciente do novo formato de metadados.

    
ui.browser.metadata.descriptive.types = dc
ui.browser.metadata.descriptive.types = ead_3
ui.browser.metadata.descriptive.types = ead_2002

Internacionalização de texto

Para que seu novo esquema de metadados seja bem integrado, você deve fornecer informações de internacionalização (i18n) para que a RODA saiba como exibir as informações necessárias na interface de utilizador da melhor maneira possível.

Edite o ficheiro [RODA_HOME]/config/i18n/ServerMessages.properties e adicione as seguintes entradas conforme necessário, assegurando-se de que a última parte da chave corresponde ao código fornecido no [RODA_HOME]/Config/roda-wui.properties descrito na secção anterior: 

    
ui.browse.metadata.descriptive.type.dc=Dublin Core
ui.browse.metadata.descriptive.type.ead.3=Encoded Archival Description 3
ui.browse.metadata.descriptive.type.ead.2002=Encoded Archival Description 2002

Finalmente, deve-se fornecer traduções para os nomes dos campos a serem processados pelo RODA através do ficheiro de configuração de visualização. Para isso, é preciso editar o arquivo [RODA_HOME]/config/i18n/ServerMessages.properties e adicionar      as seguintes entradas conforme necessário, certificando-se de que a última parte da chave corresponde ao xsl:params incluído no mapa de visualização.

O exemplo a seguir descreve como os nomes de campo no exemplo Simple Dublin Core devem ser exibidos na interface de utilizador. 

    
crosswalks.dissemination.html.dc.title=Title
crosswalks.dissemination.html.dc.description=Description
crosswalks.dissemination.html.dc.contributor=Contributor
crosswalks.dissemination.html.dc.coverage=Coverage
crosswalks.dissemination.html.dc.creator=Creator
crosswalks.dissemination.html.dc.date=Date
crosswalks.dissemination.html.dc.format=Format
crosswalks.dissemination.html.dc.identifier=Identifier
crosswalks.dissemination.html.dc.language=Language
crosswalks.dissemination.html.dc.publisher=Publisher
crosswalks.dissemination.html.dc.relation=Relation
crosswalks.dissemination.html.dc.rights=Rights
crosswalks.dissemination.html.dc.source=Source
crosswalks.dissemination.html.dc.rights=Subject
crosswalks.dissemination.html.dc.type=Type

<xsl:param name="i18n.title" />
<xsl:param name="i18n.description" />
<xsl:param name="i18n.contributor" />
<xsl:param name="i18n.coverage" />
<xsl:param name="i18n.creator" />
<xsl:param name="i18n.date" />
<xsl:param name="i18n.format" />
<xsl:param name="i18n.identifier" />

<xsl:param name="i18n.rights" />
<xsl:param name="i18n.source" />

<xsl:param name="i18n.type" />

Recarregar configuração

Depois de alterar os arquivos de configuração, deve-se reiniciar o RODA para que suas alterações se tornem efetivas. Pode-se fazer isso reiniciando o container ou o servidor de aplicação.
